# Design Doc Generator

根据用户提供的需求信息，生成结构化的AI辅助开发设计文档（prompt文档）。

## 文档结构总览

```
重点文件
├── 主要修改 → 本次代码修改主要涉及的项目代码位置
└── 参考实现 → 与当前需求相似的代码逻辑，供参考

任务背景
├── 需求背景 → 辅助AI理解需求的业务原因
├── 需求方案 → 实际需要AI去完成的任务（尽可能详细明确）
└── 注意事项 → 需求方案的补充，作为AI修改代码的边界

我的情况（自动生成）

你的情况（自动生成）

你的任务（根据模板类型自动选择）
```

## 模板选择逻辑（动态扫描）

模板存放在 `references/` 目录下，文件名格式为 `template-*.md`。

### 动态发现流程

1. **扫描目录** — 列出 `references/` 下所有 `template-*.md` 文件
2. **读取元信息** — 每个模板文件顶部包含 YAML frontmatter，声明：
   - `template_name`: 模板名称（如 "标准开发任务模板"）
   - `applies_to`: 适用场景描述（如 "一般的功能开发需求"）
   - `keywords`: 触发关键词列表（如 `[新功能, 开发, 实现, 新增, 添加]`）
3. **匹配意图** — 将用户prompt与每个模板的 `keywords` 进行匹配，选择最佳模板
4. **兜底策略** — 当无法明确匹配时，默认选择 `template-standard.md`；如果用户明确指定了模板名称，直接使用

### 内置模板

当前内置以下模板（均可被替换或扩展）：

| 模板文件 | 名称 | 触发关键词 |
|---|---|---|
| `template-standard.md` | 标准开发任务模板 | 新功能、开发、实现、新增、添加 |
| `template-bugfix.md` | Bug修复任务模板 | 修复、bug、问题、报错、异常、故障 |
| `template-review.md` | 代码Review/分析模板 | 分析、理解、讲解、学习、review、阅读 |
| `template-refactor.md` | 重构任务模板 | 重构、优化、重写、改善、性能 |

### 添加自定义模板

在 `references/` 目录下新建 `template-<自定义名称>.md` 文件，按以下格式编写：

```markdown
---
template_name: 你的模板名称
applies_to: 适用场景描述
keywords: [关键词1, 关键词2, 关键词3]
---

# 模板标题

适用于 xxx。

## 模板内容

（此处编写 "你的任务" 部分的具体步骤模板，使用 [占位符] 标记需要填充的内容）

## 使用说明

（此处编写模板的使用注意事项）
```

保存后无需其他配置，skill 会自动发现并使用新模板。

## 各模块生成规则

### 1. 重点文件

- 如果用户prompt中**提供了**主要修改的文件路径和参考实现，直接使用用户提供的内容
- 如果用户**未提供**，根据任务背景分析项目结构，推断可能涉及的文件，并标注`（推测，请确认）`
- 格式：

```markdown
# 重点文件

## 主要修改
- `path/to/file1.java` — 说明修改原因
- `path/to/file2.vue` — 说明修改原因

## 参考实现
- `path/to/reference.java` — 说明参考价值
```

### 2. 任务背景

- 从用户的prompt中提取需求背景、需求方案和注意事项
- 如果用户只给了简短描述，智能展开补充，但核心诉求保持用户原文
- 格式：

```markdown
# 任务背景

## 需求背景
[从用户prompt提取或推断]

## 需求方案
[用户明确提供的方案，或根据背景生成的详细方案]

## 注意事项
[从用户prompt提取的约束条件，或根据经验补充的注意事项]
```

### 3. 我的情况（自动生成）

根据项目技术栈自动生成，默认内容：

```markdown
# 我的情况

我是一个刚刚入职的只有一年开发经验的java后端开发，对当前业务系统不了解，且很多术语不清楚，请你详细讲解代码的时候，遇到各种术语都通过补充来详细的介绍。
```

> 如果用户在prompt中明确指定了自己的情况，则使用用户提供的内容替换默认值。

### 4. 你的情况（自动生成）

根据项目技术栈自动生成，默认内容：

```markdown
# 你的情况

你是有多年java开发经验的程序员，且对当前业务系统非常熟悉，你耐心且细心的写详细的文档，且编码能力极强。你熟悉 Spring Boot、Vue.js、MyBatis 等技术栈，能够进行前后端全栈开发。
```

> 如果用户在prompt中明确指定了AI的角色定位，则使用用户提供的内容替换默认值。

### 5. 你的任务

根据选中的模板自动生成。具体流程：

1. 扫描 `references/` 目录下所有 `template-*.md` 文件
2. 读取每个模板的 YAML frontmatter 获取 `template_name`、`applies_to`、`keywords`
3. 根据用户意图匹配最佳模板并读取其「模板内容」部分
4. 将模板中的 `[占位符]` 替换为用户需求中的实际内容（如指定目录、指定代码方法等）

> 如果用户需要查看可用模板列表，列出 `references/` 中所有 `template-*.md` 的 `template_name` 和 `applies_to` 供选择。

## 工作流程

1. **解析用户prompt** — 提取重点文件、任务背景、个人情况等信息
2. **扫描可用模板** — 列出 `references/` 目录下所有 `template-*.md` 文件，读取其 YAML frontmatter
3. **匹配模板** — 根据用户prompt中的关键词与模板 `keywords` 进行匹配，选择最佳模板
4. **读取模板内容** — 加载匹配到的模板文件中的「模板内容」部分
5. **组装文档** — 按5大模块结构组装完整文档，填充占位符
6. **输出文档** — 生成 markdown 格式的设计文档，输出到用户指定位置或直接展示

## 输出约束

- 文档语言：中文
- 格式：Markdown
- 模板中 `[占位符]` 必须根据用户需求替换为实际内容
- 如果用户提供的信息不足以填充某个模块，需要**主动询问**而非留空
- 在"你的任务"末尾始终附加注意事项：`请先生成设计文档，在我确定设计文档无问题后再生成代码`
